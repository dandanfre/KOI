Changes made to website

The changes that have been made are listed down below. The descriptions are divided into changes regarding the four html-files we have implemented, followed by changes that have been applied to the JavaScript-files, and finally, major changes in the CSS.

Index.html

Under the "price" section, at the bottom of index.html, we added a description for every treatment listed. In addition to the description, buttons were added to each treatment to fullfill the functional-requirement for available Call-to-action-buttons at each page. The arrow below the KOI-logo at the top of index.html, ended up being static and consists off one arrow. Out of fear for users mistaking underlined headers for being buttons during the usability-testing, we styled h3 element from being underlined to having a font-weight of 600, so users do not make such mistakes. The update was applied to the h3 elements in the section for treatment categories, and the price list at the bottom. Lastly, we swapped the pictures in "Our Treatments" section. In P2 the picture was a bare back with stones on it. Now the picture has been changed to hands with oil.

About_us.html

This page does not have five masseuses as planned in the section “Our masseuses”, we have only four. This because it looks better with symmetry and gives the user some sense of order and clean design. We also deleted the "Our room" caption because this was unnecessary when we have "Our parlor" as a headline.

Contact_us.html

Not many changes were made to this site. But we chose to make examples of what the user should write in the input fields with a light grey color to make it intuitively and easy to understand. We changed the "Text" input to "Message" because this is what the user actully is going to write in. We also added a green line around the input fields when typing in correct information.

Book.html

The major changes we did to the booking form was using small arrows instead of the big ones that are displayed in the prototype for book.html, in P2. In addition, placeholder text was added for the input- and select-fields to signify the user about format of the input. Placeholders have been implemented in order to assist users filling out the correct information in the form. Lastly we added green-border effects to signify correct input. This way users can go on to the next field. The implementation of red borders for wrong input have been discarded due to the fact that the correct format of input information have aldready been provided by placeholders, keeping the information to a minimum, but yet effective. The final version of the book.html landed on the MVP-version of the form. The ideal-version will have to wait for a revisit in the future.


General changes to JavaScript

In P2 we listed 6 different js-applications for maintaining a functional website, where some off them were discarded for the website. We did this due to an easier and more effective implementation in CSS. The javascript-functions that were discarded are the:

  • I. A JavaScript function to make the video on our landing page fade in when the user enters the page.
  • II. A JavaScript function to make a micro animation of arrow pointing downwards in the index page to signify that there is more content on the page

Changes to 1st required book_feedback.js: The JavaScript function for a fake "booking submitted”-message" when booking a treatment have not gone through major changes. The only time this function is triggered is when the user have filled out all the required fields (all fields) of the forms. The same function, but with another name was implemented for the contact-form in contact_us.html, and with another message.

Changes to 2nd required index_vid_effect.js: As mentioned above, this was written in CSS. Therefor the js function has been discarded.

Changes to 3rd required anim_index_arrow.js: As mentioned above, this was written in CSS. Therefor the js function has been discarded.

Changes to 4th required gallery_carousel.js: No changes were made, because our initial plan worked.

Changes to 5th pic_hover_effect.js: No changes were made, because our initial plan worked.

Changes to 6th required arrow_hidden_box.js: A JavaScript function to collaps and expand box-element with information in the our treatments segment. This javascript-function was listed as the 6th required .js. It has been changed to be the 4th required .js. In addition to the implementation of the function in JavaScript, some additional html and css-styling was necessary in order to realise the js-function. A certain div was filled out in html and styled to be displayed as none. Clicking on the button with the picture of an arrow, resulted in displaying the div with its contents(elements).

Additional JavaScript-file map.js and miniMap.js: Google provides a procedure for implementing their Google Maps API. This js-function was therefore not mentioned in the section for required js, since writing our own js-code for this implementation seemed undoable. As for the working code, our implementation of the Google Maps API works only for Google Chrome, but not for every web browsers, e.g. Mozilla Firefox, Safari etc. The given requirement stated the website should be working in Google Chrome, hence we chose not to do any further implementation, and we did not have the capacity to do more. miniMap.js works the same way as map.js, but is for rendering Google Maps API in the footer-section.

Additional JavaScript-file contact_feedback.js: We added a "message-sent"-message when the user has sent in a message, to let them know it was successfully sent. This is similar to the message that comes up when you have booked a treatment.

Modularity Requirement: We fulfilled the modularity requirements by using JavaScript. navbar_footer_modularity.js was used to make the navbar(id="header") and the footer(id="footer"). This was included in each page in one div for the navbar and one div for the footer. As for the design of navbar and footer, the navbar did not go through any major changes, other than not implementing underline for active site. The reason is because we felt this would be more of a distraction, than a helpful navigation signifier, but we will test this hypothesis during user-testing. Footer: We made the text of the content thinner(weight: 200) and smaller(font-size: 20px) to make it fit better in the visual hierarchy. We also included a “map” header above the map to improve the universal design of the page. We made one stylesheet named style.css that was used to style the whole website.
