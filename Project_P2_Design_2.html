The client

Client name: KOI

KOI is a fictive massage business based in Trondheim. The name KOI derives from the well known carp fish, koi. The fish is often associated with strength of character and perseverance, and KOI wants to make the customers feel the same after a visit. KOI is striving to help individual customers feel relaxed, revitalized and strong. The business’ main focus is private, individual customers, but KOI also offers treatment for businesses that want to include KOI’s services in their employee benefits.

Purpose and goals

Many massaging businesses have a static and sterile website. KOI wants to be represented in a professional manner on digital platforms reflecting the same zen feeling customers acquire through KOI's treatment. At the same time the website is informative about the different treatments KOI offers and therefore segments clearly between offers meant for private customers, and businesses (see subsection "Business" under "Our Treatments"), thus giving a trustworthy display and easy understanding. In addition, the website should provide customers with the opportunity to provide action, such as ease of booking and a convenient way to communicate with KOI (see "Contact Us").

Audience

The website's audience is divided into both existing and new customers, and segmented between private individuals and businesses. Existing customers who know how the site works and know what treatment they want, can jump quickly to the booking- or contact-form within two clicks. As for new customers they will have the opportunity to explore what treatments KOI offers, both private individuals and businesses alike. The typical customer is in the age group of 25-35, working full-time, quality conscious and comfortable economically. They use massage for the purpose of well-being, getting a break from a hectic day-to-day life and stress, not to treat injury. For businesses it should be easy for them to find KOI's package offers meant for them, so they can provide services as part of their employee benefits.

Navigation Structure

We want our website to be fluent and repsonsive when in use, so that our policy about segmenting between new and regular users remains. Regular users usually know what treatment they want, and since the navigation bar is located at the top of each webpage with a distinct "Book"-button it is easier for this type of users to go head straigth to the booking form, i.e. "Book"- webpage. As for the new users, the navbar at top helps with creating a mental model of how KOI's website is structured, in addition to orientation. Having a navigation bar applies to all of KOI's users making the website more intuitive to use, at the same time helping to avoid annoyances when using it.

Minimum Requirements

We will have 5 different JavaScript-applications for maintaining a functional website. These will have different difficulties in the implementation of the website, and we believe that each and one of the member is able to code the given task. We will be fulfilling the requirements of JavaScript applications in this way:

  1. A JavaScript function for a fake “booking submitted”-message when ordering a treatment.
  2. A JavaScript function to make the video on our landing page fade in when the user enters the page.
  3. A JavaScript function to make a micro animation of arrow pointing downwards in the index page to signify that there is more content on the page.
  4. A JavaScript function to make an automated picture carousel for the “about us” page.
  5. A JavaScript function to make the pictures of masseuses change upon hovering.
  6. A JavaScript function to collaps and expand box-element with information in the our treatments segment.

Page Modularity

In the 3rd part of the project(i.e. P3) we have to support modularity when coding KOI's website. This is in order to avoid writing the same code several times. Repeated items or code snippets will be handled by using external files of JavaScript code. The JS code will be employed to either include and/ or create repeating element(s) for the relevant html-pages. The plan for modularity is as follows:

  • index.html loads "navbar_footer_modularity.js" The script creates the NAV-bar and footer form pre-configured objects, where the links will redirect the users to the proper page
  • Using external JavaScript files - no inline scripting.
  • External CSS style sheet - no seperated style files for each pages.
